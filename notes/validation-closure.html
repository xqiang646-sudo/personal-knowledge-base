从候选结果到正式结论

“这次成功了”是一条观察；“问题已关闭”是一项需要环境、场景、指标、回归和可复现证据共同支持的结论。

贡献角色：导师原创整理　对象：工程验证与问题交接入门者　证据状态：通用模板；实际验收标准由项目负责人定义

安全范围：公开记录使用合成示例和脱敏路径，不上传原始车辆数据、日志、地图、账号或内部链接。

状态环境边界场景与指标证据包混合结果回归交接模板练习

先给结论一个明确状态

不同状态允许表达的结论
状态含义可以怎样写
未验证只有现象、想法或未执行的修改“待在目标环境检查”
候选在有限样本或替代环境出现正向结果“候选修复；已通过本地样例，尚未目标验收”
正式在指定目标环境按预先约定的条件通过“在版本、场景和指标 X 下正式通过”
已失效环境、版本或新证据使旧结论不再适用“旧结论仅适用于先前版本，需重新验证”

状态不是质量评价：候选结果可以很有价值，但不能省略它尚未覆盖的环境和风险。

环境边界：在哪里通过，就只证明到哪里

开发机

适合语法、单元测试、静态检查和快速样例；不能自动代表目标系统的依赖、硬件或时序。

仿真环境

适合可重复场景和边界输入；传感器噪声、执行器、网络和真实负载仍可能不同。

目标计算平台

可验证部署依赖、运行时、资源和吞吐；没有真实场景时仍不能替代完整系统验收。

受控现场 / 实车

只能在授权的安全流程、场地、负责人和停止条件下进行，并完整保留条件与例外。

例如，本地 Mac 上的构建成功不能写成目标 Ubuntu 主机正式通过；目标主机一次启动成功，也不能自动证明实车功能达标。

先定义场景和通过标准，再运行测试

  1. 环境：主机、系统、运行时、硬件、代码、模型和配置版本。
  2. 前置条件：依赖服务、输入数据、地图、路线和系统状态。
  3. 场景：输入怎样变化，持续多久，哪些边界条件必须覆盖。
  4. 指标：正确性、频率、时延、资源、稳定性或安全约束怎样计算。
  5. 门槛：什么叫通过、失败、无效样本和允许例外。
  6. 停止条件：何时立即停止测试并转入安全处置。

测试完成后才临时挑选有利指标，会让结论无法审计。通过标准应在看到结果之前就写清。

混合结果不能压成一个“通过”

分别记录

按场景、版本、输入类型或运行时长列出通过、失败和未执行项。

保留失败

失败样本是适用范围的边界，不应因为多数样本通过就被删除。

区分原因

测试设施失败、输入无效和被测功能失败是不同结论，需要不同后续行动。

限制措辞

写“场景 A 通过、场景 B 失败”，不要写“整体正常”掩盖差异。

关闭问题前的回归闭环

复现原问题
→
固定基线
→
最小修改
→
同条件复测
→
相关场景回归
  • 原问题在修改前能够稳定复现，避免“问题自己消失”。
  • 修改后用相同输入、环境和指标复测，保证比较公平。
  • 检查相邻功能和过去失败场景，避免修好一处、破坏另一处。
  • 记录新版本与旧版本差异、回滚方法和需持续观察的风险。

实习生练习

  1. 把“我本地跑通了，所以已经修好”改写成准确的候选结论。
  2. 为一个虚构的频率下降问题定义场景、指标、门槛和无效样本。
  3. 给出一个“多数通过但不能整体判定通过”的混合结果例子。
  4. 用交接模板写一份不超过一页的可复现问题记录。
