感知算法与地图基础

感知把像素和空间点转换成可供规划使用的目标；地图提供静态环境、道路拓扑和交通规则。

Point Cloud：点云是什么

激光雷达根据激光往返时间计算距离，再结合光束方向生成三维点。一个点常写成：

\[ \mathbf p = \left(x,\,y,\,z,\,\mathrm{intensity},\,\mathrm{ring},\,\mathrm{timestamp},\ldots\right) \]

点本身只说明“这个空间位置有反射”，并不知道它属于车辆、行人、地面还是墙。

实时点云

车辆当前扫描到的动态环境，用于检测、地面分割、占据栅格和定位匹配。

Point Cloud Map

提前采集并拼接的静态三维环境，常保存为 PCD，主要用于 NDT/ICP 等定位。

常见预处理

  • 多雷达时间同步、拼接和统一坐标系。
  • 运动畸变校正：补偿雷达扫描期间车辆已经移动造成的形变。
  • 裁剪车身反射、感兴趣区域和无意义远距离点。
  • 雨雾、飞虫、孤立点过滤和体素降采样。
  • 地面分割、聚类、体素化与深度学习特征提取。

大量点执行相同变换、体素化和神经网络计算时适合 GPU；简单裁剪、格式转换和部分传统聚类也可能运行在 CPU。

项目记录边界：材料中提到彩色点云与白色点云的高斯拟合及一组定位质量数值，但没有给出指标名称、单位和算法定义。它们只能作为特定项目的调试记录，不能当作通用定位阈值。

三类地图解决不同问题

地图主要信息主要用途
Point Cloud Map静态环境三维表面LiDAR 地图匹配与定位。
Vector / Lanelet2 Map车道几何、拓扑、限速、停止线、交通灯和规则路线、行为规划、交通规则和感知 ROI。
Occupancy Grid网格被障碍物占据的概率自由空间判断、避障和低速规划。

Vector Map 与 Lanelet2

Vector Map 是“用点、线、多边形和语义描述道路”的地图类型；Lanelet2 是一种具体的数据模型、文件格式和代码库。

标量只有大小，例如速率、质量和温度；矢量同时包含大小与方向，例如速度、位移和加速度。“矢量地图”中的矢量更偏计算机图形含义：地图由坐标点、线、曲线和多边形表示，而不是由像素组成。

一个 lanelet 通常由左右边界组成，并可以关联中心线、限速、转向方向、停止线、交通灯、优先权、前驱/后继和是否允许变道。

不能只用中心线：规划还需要知道车道宽度、边界能否跨越、路口冲突区域、停止线以及交通灯控制关系。

.osm 与 XML

.osm 是基于 XML 的结构化文本文件，不是图片。Lanelet2 借用 OSM 的组织形式保存点、线和关系；普通 OpenStreetMap 数据不能直接等同于自动驾驶高精地图。

<node id="1" lat="..." lon="..." />
<way id="100">...</way>
<relation id="200">
  <member type="way" ref="100" role="left" />
  <member type="way" ref="101" role="right" />
  <tag k="type" v="lanelet" />
</relation>

地图文件负责保存数据，Lanelet2 代码库负责读取和查询，RViz 或地图编辑器负责把数据画出来。

  • Detection：当前这一帧看到了什么，输出类别、位置、尺寸和置信度。
  • Fusion：把图像语义、LiDAR 几何和 Radar 速度等信息结合。
  • Tracking：判断连续帧是否为同一目标，输出稳定 ID、速度和加速度。
  • Prediction：结合历史状态和地图，估计未来几秒的多条可能路径与概率。
  • Occupancy：不强调目标类别，只回答某块空间是否可能被占据。

YOLOX：图像二维目标检测

YOLOX 的输入是 RGB 图像，输出二维检测框、类别和置信度。它通过卷积与大量矩阵运算逐级提取边缘、纹理和高级语义特征。

图像 → 特征提取 → 分类与框回归 → 2D box + class + confidence

YOLOX 采用 anchor-free 思路，并将分类与回归头解耦。GPU 能同时处理大量像素和特征图，使实时检测成为可能。

局限：普通单目二维检测通常不能单独给出可靠的真实三维距离、尺寸和姿态，需要深度、标定、LiDAR 或 BEV 感知补充。

BEVFusion：在鸟瞰空间融合相机与 LiDAR

BEV 是 Bird's-Eye View。相机语义丰富但深度不明确，LiDAR 几何准确但稀疏；BEVFusion 先把两种特征变换到统一的车辆鸟瞰坐标系，再融合并进行三维检测。

多相机图像 → 图像骨干网络 → 视角变换 → Camera BEV ┐
                                                     ├→ BEV Fusion → 3D检测
激光点云     → 体素化/编码器              → LiDAR BEV ┘

它通常需要同时处理多路高分辨率图像、几十万点的点云、坐标变换、特征融合和三维检测头，因此比单路图像检测更依赖 GPU 与显存。

常见部署链路是 PyTorch → ONNX → TensorRT Engine。FP16 可降低显存并利用 Tensor Core，INT8 进一步提速但通常需要量化校准并验证精度。

RViz 也使用 GPU，但主要是渲染点云、地图和车辆模型，不等于感知网络正在高效推理。画面卡顿和算法卡顿必须分别测量。

感知问题的排查顺序

  1. 时间同步：图像、点云和定位是否属于正确时刻。
  2. 内外参标定：相机内参、相机—LiDAR 外参是否正确。
  3. TF 与坐标轴：sensor、base_link、odom、map 的方向和变换是否正确。
  4. 数据预处理：点字段、畸变校正、裁剪、滤波、单位是否正确。
  5. 模型推理：权重、输入尺寸、精度模式、GPU 和后处理是否匹配。
  6. 阈值与融合：置信度、NMS、关联门限、时间窗口和地图 ROI 是否合理。

工程经验：检测框整体偏移或融合错位时，优先检查时间、标定和坐标系，而不是立刻重新训练模型。
