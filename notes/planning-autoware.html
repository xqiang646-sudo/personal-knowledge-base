规划控制与 Autoware

规划不是一个单独算法，而是从全局路线、驾驶场景、行为决策到可执行轨迹的逐层约束过程。

Route / Mission Planner

根据起点、终点和 Lanelet2 路由图选择应经过的道路与车道段。输出是全局路线，不是车辆可以直接跟踪的平滑轨迹。

Scenario Planner / Selector

选择完全不同的规划模式，例如正常道路行驶与停车场自由空间行驶。它回答“现在应该采用哪套规划流程”。

Behavior Planner

决定车辆应该做什么，例如保持车道、换道、绕开静态障碍物、从路肩起步、靠边停车、红灯停车或在人行横道前减速。

Behavior Path

主要处理横向和道路拓扑：跟车道、换道、侧移、绕障、起步和靠边停车。

Behavior Velocity

主要在路径上施加交通规则速度：停止线、交通灯、路口、人行横道、盲区和减速带。

Motion Planner

把行为决策变成连续、平滑、无碰撞且车辆可实现的轨迹，确定每个轨迹点的位置、朝向、曲率、速度、加速度和 jerk。

  • 满足道路边界、障碍物和车辆几何约束。
  • 避免曲率、横向加速度和 jerk 过大。
  • 与上一周期轨迹保持连续，避免控制指令剧烈跳变。
  • 可采用优化、采样、Frenet、Hybrid A* 或自由空间规划等方法。

Path 与 Trajectory 的区别

Path：从哪里经过

主要表达空间几何，如 x、y、yaw，可按沿路径距离 s 参数化。

Trajectory：何时以多快经过

在路径上加入时间和运动状态，如 v、a、curvature。

Autoware 的具体消息中 Path 也可能携带速度字段，但“Path 偏几何、Trajectory 偏时序运动”的概念区分仍然重要。

Control：把轨迹误差变成执行命令

控制器持续比较目标轨迹和车辆实际状态，计算横向位置误差、航向误差和速度误差。

横向控制

MPC、Pure Pursuit 等方法输出方向盘角度和角速度。

纵向控制

PID 或加速度控制输出目标加速度、驱动和制动请求。

控制命令经过命令门控和 Vehicle Interface，转换成具体底盘的 CAN 报文。AEB 等安全层会在预测到碰撞风险时触发额外制动。

输出规划

地图提供交通灯大致位置，相机和标定把地图对象投影到图像，检测器精确定位，分类器判断状态，多相机融合与仲裁生成最终结果，再交给行为规划决定停车或通行。

Autoware 的重要思想：地图提供先验，神经网络处理观测，二者相互约束。

系统安全、故障诊断与评估

自动驾驶不仅处理正常情况，还要处理点云超时、定位漂移、GPU 过热、节点退出、网络异常、轨迹非法和控制命令异常。

  • Diagnostics：汇总节点和硬件状态，形成故障关系。
  • Minimum Risk Maneuver：在能力不足时执行减速、靠边或停车等最小风险动作。
  • Trajectory Validator：检查碰撞、曲率、速度、加速度和轨迹跳变。
  • Evaluator：计算偏差、曲率、jerk、TTC、稳定性和异常停车等指标。

调试时应沿数据链逐级确认 Publisher、Subscription、话题频率、时间戳、frame_id 和延迟，而不是只看 RViz 画面是否流畅。
