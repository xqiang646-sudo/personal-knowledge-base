先理解 nuScenes 是什么

nuScenes 是自动驾驶多模态时序数据集。官方 v1.0 包含 1000 个约 20 秒的城市驾驶场景，采集于波士顿和新加坡；传感器套件包括 6 个相机、5 个毫米波雷达、1 个激光雷达以及 GPS/IMU。

传感器文件
+
关系型元数据
+
标定 / Ego Pose
+
标注 / 地图 / 评测

因此它不是“一堆图片和点云文件”。数据文件回答“测到了什么”，元数据回答“何时、由哪个传感器、在什么位姿和坐标系下测到，以及它与标注怎样关联”。

核心：它是一张由 token 连接的数据图

scene
  ├─ log ── location / map
  └─ sample ── prev / next
       ├─ sample_data ── sensor file
       │    ├─ calibrated_sensor ── sensor → ego 外参、相机内参
       │    └─ ego_pose ── ego → global 位姿
       └─ sample_annotation
            ├─ instance
            └─ category / attribute / visibility
先分清逻辑时刻、传感器帧和标注属于哪一层
记录表示什么数据任务中常用字段
scene从一段 log 中抽取的连续场景首尾 sample、log、场景内 sample 数量
sample2 Hz 的多传感器标注关键时刻，不是磁盘文件prev/next、通道索引、annotation 索引
sample_data某个传感器在某一时刻的一份真实数据文件、时间戳、关键帧标志、标定与 ego pose token
calibrated_sensor传感器相对 ego body frame 的标定外参；相机还记录内参
ego_pose该传感器时刻的 ego 相对地图 global frame 位姿平移、四元数、时间戳
sample_annotation某个 sample 上一个目标的 3D 标注instance、类别、属性、可见性、尺寸与传感器点数

token 是记录主键，*_token 像外键一样连接其他表。devkit 加载后会为 sample["data"] 和 sample["anns"] 建立反向索引；学习者仍应知道它们连接到哪些原始表。

从数据根目录生成可查询索引

目录只描述文件放在哪里，索引才描述任务样本是什么。先盘点当前加载对象，不要从官网抄一个总数当作本地结果。

<NUSCENES_ROOT>/
├── samples/       # 关键帧传感器数据
├── sweeps/        # 关键帧之间的中间传感器数据
├── maps/          # 地图数据
└── v1.0-mini/     # JSON 元数据；也可能是 trainval / test

inventory.json

记录 dataset version、已安装组件、各表实际行数、channel、modality、location，以及条款核对日期。

samples.csv

至少包含 version、split、scene/sample/sample_data token、sample 与 sensor timestamp、channel、文件、keyframe、标定和 ego pose token。

annotations.csv

至少包含 annotation/sample/instance token、category、attribute、visibility、translation、size、rotation 和传感器点数。

为什么分两张表

一个 sample 可连接多通道和多个 annotation。全部展开成一张巨表会产生大量重复，也更容易误算分母。

from pathlib import Path

rows, missing = [], []
for scene in nusc.scene:
    token, visited = scene["first_sample_token"], set()
    while token:
        if token in visited:
            raise RuntimeError(f"cycle in sample chain: {token}")
        visited.add(token)
        sample = nusc.get("sample", token)
        for channel, sd_token in sample["data"].items():
            sd = nusc.get("sample_data", sd_token)
            path = Path(nusc.dataroot) / sd["filename"]
            rows.append((scene["token"], token, channel, sd["timestamp"], str(path)))
            if not path.is_file():
                missing.append(sd_token)
        token = sample["next"]

一段 20 秒 scene 通常约有 40 个 sample，但代码必须读取 scene["nbr_samples"] 并与实际遍历数核对。不同传感器的 timestamp 不能强行替换成 sample timestamp。

质量门：文件、关系、时间、标定和标签

报告格式统一为：检查项 / 分母 / 通过 / 警告 / 失败 / 代表 token
层级检查示例不能草率下的结论
文件层sample_data.filename 是否存在、可读取、类型符合预期文件存在不等于内容和语义正确
关系层外键 token 能否解析；scene 的 prev/next 链是否完整、无环不能静默跳过断链记录
时间层每条链是否单调；通道间时间偏差分布；sweep 是否保留自身时刻传感器时间不同不自动等于坏数据
标定层外参、相机内参、ego pose 是否存在且数值可用引用存在不等于坐标方向使用正确
标签层类别、属性、可见性、尺寸、点数和前后 annotation 是否可解释num_lidar_pts=0 不能一律判作坏标注

质量结论必须分层：schema/文件损坏、合法但值得关注的数据、任务评测过滤规则、官方已知问题不是同一件事。报告应保留代表 token，并链接当前官方规则，而不是靠截图和印象解释。

坐标数据合同：先声明字段在哪个 frame

sensor frame
→
ego frame at sensor time
→
global frame of map
  • calibrated_sensor 提供 sensor 到 ego 的外参；ego_pose 提供该数据时刻 ego 到 global 的位姿。
  • 原始 sample_annotation 位于 global frame；devkit 的 get_sample_data() 会把 box 转到当前 sensor frame。
  • 标注尺寸顺序是 width, length, height；记录中的四元数顺序是 w, x, y, z。
  • 派生标签和统计必须记录源 frame、目标 frame、变换链、时刻和代码版本。

矩阵推导详见前置课程 M06 坐标系转换。本课继续养成习惯：任何 point、box、pose 或 velocity 出现时，都先写清它当前属于哪个坐标系。

任务合同：切分、标签空间与输出格式

数据准备不是“把文件拷过去”。它要冻结任务输入、split、标签映射、过滤规则、坐标与输出协议。

相同数据集可服务不同任务，标签空间不能混用
概念官方 v1.0 边界课程要求
论文中的 3D box taxonomy23 类，另有 8 种属性作为源 annotation 语义理解
Detection benchmark合并和过滤后使用 10 个检测类记录从源类别到任务类别的映射
nuScenes-lidarseg32 个通用语义类，挑战合并为 16 类作为独立标签空间，不能和 detection 混算
官方 splittrain/val/test 与 mini_train/mini_val 均按 scene 组织调用官方 split 工具并验证 scene 无重叠
test提供传感器数据但不公开 annotation不能把无标签误判为本地数据损坏
派生 2D box可由 3D box 投影生成，但不等同人工紧致 2D 真值保留源 annotation token、算法、边界裁剪和遮挡限制

模型能输出 box 不等于结果可评测。Detection 提交还要满足 global 坐标、类别、速度、属性、JSON 结构与覆盖评测 sample token 等协议。

五个循序实验

  1. 本地数据体检单：加载 v1.0-mini，统计表、scene、channel、location 和组件；交付 inventory.json，所有数字必须来自当前对象。
  2. 建立两张任务索引：生成 sample 与 annotation manifests，沿每个 scene 的 next 遍历，核对 nbr_samples、文件和外键。
  3. 制作 QC 报告：执行文件、引用、时间、标定和标签检查；每个 mini scene 用确定规则选择一个 sample 渲染，并把自动检查与人工观察分开。
  4. 建立切分与标签合同：用官方 API 获取 mini_train/mini_val，验证 scene 无交集；以 detection 为例记录源类别到任务类别映射，并视觉核对少量派生 2D box。
  5. 生成可交付数据卡：汇总 inventory、manifest、QC、split、统计与可视化；生成覆盖 mini-val sample token 的空 prediction JSON 做格式 dry-run。

常见误区

“sample 就是一帧图片”

它是多传感器标注时刻；每份真实文件属于 sample_data。

“所有通道完全同步”

应比较各自 timestamp；跨时刻融合还要补偿 ego motion。

“随机切帧更灵活”

相邻时序帧高度相关；优先使用官方 scene-level split，避免泄漏。

“23 类都参加检测评测”

源 taxonomy、detection 与 lidarseg 是不同标签空间。

“投影 2D box 就是人工真值”

投影框可能不紧致，还受遮挡和画面边界影响，必须标记为派生标签。

“mini 指标高说明模型有效”

mini 用于学习与集成调试，正式结论需官方 split 和评测协议。
