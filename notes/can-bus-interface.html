CAN 总线接口有什么用

CAN 总线是车辆控制器之间交换短消息的通信网络。自动驾驶计算机通过 CAN 接口发送转向、油门和制动指令，同时接收车速、方向盘角度、挡位和故障状态。

CAN 传输的是一帧一帧的报文，不是文件。一帧通常包含：

  • CAN ID：消息类型和总线仲裁优先级；数值越小，优先级通常越高。
  • Data：真正的数据字节。
  • DLC：数据长度。
  • 时间戳：报文接收或发送的时间。
CAN ID：0x201
数据：08 64 00 15 00 00 00 00

物理连接的三个重点

  1. CAN_H 接 CAN_H，CAN_L 接 CAN_L，并确保设备共地。
  2. 所有设备波特率一致。常见值包括 125、250、500 kbps 和 1 Mbps。
  3. 总线最远两端各有一个 120Ω 终端电阻。断电测量 CAN_H 与 CAN_L，正常通常约为 60Ω。

真正理解报文需要 DBC 或通信协议

能收到数据不等于能看懂数据。要正确控制车辆，还需要知道信号所在字节、比例系数、偏移量、大小端、校验方式、计数器、发送周期和使能条件。

核心：CAN 接口只是收发通道，车辆厂商提供的 DBC 或 CAN 通信协议才决定每个字节的真实含义。

RViz 与 HMI 有什么区别

IPC@RVIZ

面向工程师调试。

显示点云、地图、目标框、坐标系、规划轨迹和算法状态，用来检查系统“看到了什么、在想什么”。

IPC@HMI

面向驾驶员或操作员。

提供启停、任务设置、模式切换、车速、故障和报警等关键信息，用来告诉系统“要做什么”。

对比项RVizHMI
主要用户开发、测试、调试人员驾驶员、操作员
主要内容点云、地图、目标、轨迹按钮、仪表、状态、报警
主要目的定位算法与数据问题安全、清晰地操作车辆

ipc@RVIZ 也可能只是 Linux 终端中的“用户名@主机名”，可用 hostname 确认。

NVIDIA GPU 在系统中做什么

CPU 擅长复杂控制逻辑和串行任务；GPU 拥有大量相对简单的计算单元，擅长让大量数据同时执行相似运算，因此适合图像、点云、矩阵计算和神经网络。

  • CUDA Core：执行通用并行运算。
  • Tensor Core：加速神经网络中的矩阵乘加。
  • 显存 VRAM：存放模型参数、图像、点云和中间特征。
  • CUDA / cuDNN / TensorRT：分别提供通用计算、深度学习算子和推理优化能力。

在自动驾驶中，YOLOX、BEVFusion 等感知模型通常使用 GPU；规划、状态机和车辆控制更多依赖 CPU。RViz 也会使用 GPU，但主要用于三维画面渲染。

CAN 通信异常排查清单

  1. 确认 CAN_H、CAN_L 没有接反，设备已经共地。
  2. 确认所有设备使用相同波特率。
  3. 断电测量 CAN_H 与 CAN_L，检查是否约为 60Ω。
  4. 确认 Linux 中正确的 CAN 通道已经启动。
  5. 使用 candump can0 检查是否收到报文。
  6. 检查错误帧、错误计数以及是否进入 bus-off。
  7. 最后核对 CAN ID、DBC、发送周期、计数器和校验规则。
sudo ip link set can0 type can bitrate 500000
sudo ip link set can0 up
candump can0
ip -details -statistics link show can0

最终总结：CAN 让控制器可靠通信；GPU 处理大规模感知计算；CPU 完成系统调度与决策；RViz 用于调试，HMI 用于操作。
