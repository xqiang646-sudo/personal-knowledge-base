沿 ROS 2 图谱排查数据链

不要从“某个算法可能坏了”开始猜。先把系统还原成发布者、消息契约、传输条件和订阅者，再证明数据在哪一段停止满足条件。

贡献角色：导师原创整理　对象：自动驾驶入门学习者　证据状态：通用方法；具体项目结论需在目标环境重新验证

安全范围：以下命令用于只读观察。不要在实车或生产系统中尝试发布控制消息、修改参数或重启节点。

系统图谱消息契约时间与 TF上下游链只读命令排查例子练习

第一层：进程存在，不等于节点工作

ROS 2 的计算由节点完成，节点通过话题、服务和动作协作。排查时要从实际运行图谱出发，而不是只看源码目录、launch 文件或终端里有没有一个进程名。

进程层

程序是否启动、是否退出、是否反复重启、使用了哪个入口和配置。

节点层

节点是否出现在当前 ROS domain，名称和命名空间是否符合预期。

图谱层

谁发布、谁订阅、端点数量是否正确，是否连接到预期的上下游。

功能层

消息内容是否有效，下游是否真正采用，场景行为是否符合预期。

ROS 2 Jazzy 官方入门文档分别解释了 ROS 2 节点和话题的基本模型；项目使用其他 ROS 2 发行版时，应切换到对应版本文档。

第二层：话题存在，不等于通信成立

发布者和订阅者必须在多个条件上同时匹配。只执行 ros2 topic list 只能证明名称被发现，不能证明下游收到了可用数据。

从名称到语义逐层检查消息契约
检查项要回答的问题典型失败
名称与命名空间上下游指向的是同一个完整话题名吗？重映射、前缀或命名空间不同
消息类型发布者与订阅者使用同一接口类型吗？接口版本或类型不一致
QoS可靠性、持久性和队列等策略兼容吗？端点可见但无法配对
频率与新鲜度是否持续发布，时间戳是否仍在更新？残留端点、卡死或频率不足
消息内容数组非空、数值范围和状态字段合理吗？有消息但内容不可用
消费结果下游回调、输出或状态是否随输入变化？收到消息但被过滤或拒绝

QoS 会影响端点能否连接和数据怎样传递；概念与兼容性应以当前发行版的官方说明为准，本页参考 ROS 2 Jazzy QoS 文档。

第三层：检查时间、坐标系与数据同步

时间戳

消息时间来自系统时钟、传感器时钟还是仿真时钟？节点是否启用了相同的时钟来源？

新鲜度

“最后一条消息”距离现在多久？一次历史消息不能证明当前数据仍在流动。

TF

消息的 frame_id 能否在对应时间变换到算法需要的目标坐标系？

同步

多传感器消息是否落在可接受的时间窗口内，队列和近似同步策略是否会丢弃数据？

判断方法：先证明消息到达，再证明时间有效，再证明坐标可变换，最后才讨论算法语义。

第四层：沿输入—处理—输出—消费者前进

传感器 / 地图 / 路线
→
预处理 / 感知
→
融合 / 预测
→
规划
→
控制输出

Autoware 官方概念说明将自动驾驶系统组织为感知、规划、控制等功能域。实际包名、接口和数据流会随版本变化，所以应从当前系统图谱确认，而不是背诵固定话题名。

每到一段，都问四个问题：输入端点是谁？输入内容有效吗？当前节点产生了什么输出？哪个下游消费者实际采用了输出？第一个无法被证据回答的问题，就是下一步检查点。

一组通用的只读检查

# 1. 当前运行图谱
ros2 node list
ros2 node info /<NODE>
ros2 topic list -t

# 2. 端点、类型与 QoS
ros2 topic info /<TOPIC> --verbose
ros2 interface show <MESSAGE_TYPE>

# 3. 频率与一条样本（必要时先限制字段或重定向到安全位置）
ros2 topic hz /<TOPIC>
ros2 topic echo /<TOPIC> --once

# 4. 坐标变换
ros2 run tf2_ros tf2_echo <TARGET_FRAME> <SOURCE_FRAME>

注意：命令输出可能包含地图位置、设备标识或其他现场信息。公开笔记只记录脱敏后的结构和结论，不粘贴原始实车消息。

虚构例子：可视化里看不到检测结果

  1. 确认显示工具订阅的完整话题名和消息类型。
  2. 确认目标话题有活动发布者，发布频率和时间戳持续变化。
  3. 抽取一条消息，检查检测数组是否为空、置信度与坐标是否合理。
  4. 确认消息 frame_id 到显示固定坐标系的 TF 在同一时间有效。
  5. 如果消息和 TF 都有效，再检查显示插件、过滤条件和队列设置。

只有在第 3 步证明消息内容异常时，才把问题范围缩到检测节点或它的输入；在此之前，“模型坏了”只是一个尚未验证的假设。

实习生练习

  1. 画出一个虚构的三节点数据链，并标出发布者、话题类型和订阅者。
  2. 设计一个“话题存在但下游无数据”的 QoS 失败例子。
  3. 解释时间戳正常但 TF 不可用时，为什么算法或可视化仍可能丢弃消息。
  4. 把一次排查过程写成“观察—假设—可证伪检查—结论”。
